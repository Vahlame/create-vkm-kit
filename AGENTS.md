## Project overview

This project documents and ships **cross-platform** tooling for **Markdown vault memory** exposed to agents through **MCP** â€” `basic-memory` for read/write/search, plus the **`obsidian-memory-hybrid`** sidecar that adds passage-first hybrid (BM25 + semantic) retrieval, vault-locked file tools and `vault_audit` (the recommended vault surface) â€” a **Go daemon** for git-backed sync, and an **initializer** CLI (`--ide codex,claude,cursor`, or `--full` for the whole stack in one shot â€” ADR-0022). ADRs in `docs/adr/` are the source of truth for design; ADR-0010â€“0015 cover the v1â†’v2 stack transition; the **v3 kit layout** (no `.ps1`/`.vbs` shipped, daemon `proc_windows.go` for silent Windows operation) is documented in `docs/legacy/v2-to-v3-script-free-kit.md`.

## Installing this kit for a user (when told "install it")

If a user hands you this repo and asks you to **install / set it up** (not contribute to it), do this â€” it's a single self-verifying entry point:

```bash
npm install        # once, to fetch the installer's deps
npm run setup      # preflight deps â†’ `--full` install â†’ verify â†’ restart notice
```

`npm run setup` ([`scripts/agent-install.mjs`](./scripts/agent-install.mjs)) **preflights** dependencies (`node`, `uv`, `git`, `python`/`pip`), **auto-detects** which agent CLIs are on PATH (`codex`, `claude`; falls back to Cursor's `mcp.json`), runs the `--full` install (hybrid + semantic + index + rules, or the basic-memory stack if Python is absent), then **verifies** (vault scaffolded, index built, `codex/claude mcp list` shows the servers) and prints a status table. Pass options after `--`: `npm run setup -- --vault "<path>" --ide codex,claude --lang en`. Use `npm run setup:dry` to preview with zero writes.

> **Hard limit â€” be honest about it:** registering an MCP server does **not** make its tools live in the _current_ session. No agent can hot-load its own MCP. After `npm run setup`, tell the user to **restart** Claude Code / Codex (or reload the Cursor window); the memory tools (`vault_hybrid_search`, â€¦) answer in the next session, not this one. Verify wiring with `claude mcp list` / `codex mcp list` and confirm before declaring success.

For the no-clone, npx-only path (basic install without hybrid), see [`docs/en/install-with-agent.md`](./docs/en/install-with-agent.md).

## Memory protocol (vault)

Use a private git vault (example layout in `examples/`):

1. **START_HERE.md** â€” entrypoint for agents.
2. **MEMORY.md** (or equivalent) â€” durable global preferences; separate facts vs hypotheses explicitly.
3. **SESSION_LOG.md** â€” short chronological decisions.
4. **PROJECTS/<name>.md** â€” per-project context; do not mix projects.

**Do not** store secrets, tokens, or literal hardware IDs. Prefer wikilinks between notes. Rotate `SESSION_LOG` when it grows beyond team policy (`obsidian-memory-rag rotate-log`).

**Recall proactively; write sparingly.** Search the vault _before answering_ (don't wait to be asked) whenever a task continues prior work, names a project/person/tool you may have notes on, repeats an earlier question, or revisits a decision â€” `vault_hybrid_search("<topic>")` and answer from the returned section. Save only what's reusable beyond the session (closed architecture, hard-won decisions, firm preferences, lessons); never per-day TODOs, command output, or what the code already documents. Dedup before writing (search first) â€” a noisy vault is one nobody reads.

**Why passage-first pays.** A `vault_hybrid_search` hit returns one section (~150â€“250 tokens); reading the whole note is often 10â€“40Ã— that. Measured on the wire (the compact JSON you actually read â€” ADR-0032/0034, CI-gated): median savings vs whole-note reads = 62% at `limit: 3`, 37% at 5, with 100% of labelled queries still answered. Default `limit` is 10; pass 3â€“5 on targeted recalls, and `explain: true` only when you need ranking diagnostics. Reach for `vault_read_file` only when the section genuinely isn't enough. Note `memory_extract_candidates` flags duplicates by **BM25 text overlap, not meaning** â€” a flagged "duplicate" may be a related-but-distinct note, so you and the human judge it; an unflagged bullet may still restate an existing idea in different words.

**Evolve and self-check.** Form hypotheses about recurring patterns â€” track tech in `STACKS/` (one note each), log likely high-impact anti-patterns in the user's code to `PRACTICES/observations.md`, and promote to `PRACTICES/confirmed-{good,bad}.md` only when the user confirms (coach by asking, never impose). Before a non-trivial answer, do a brief internal self-check (assumptions, edge cases, what would make this wrong) scaled to the task. **Adapt to the active model's strengths** via `_meta/agent-profiles.md` â€” read your row (Claude / Composer / GPT / DeepSeek / Geminiâ€¦) and record which model fits which task type so the vault learns over time. Keep it cheap â€” passage-first, terse bullets, no long monologues.

**Token-efficient reads (fan-out safe).** Prefer `vault_hybrid_search` â€” it returns the matching **section**, not the whole note â€” over a full `read_note`; never read `SESSION_LOG.md` or large `PROJECTS/*` notes whole. When spawning sub-agents, the orchestrator distills context **once** and passes the excerpt in each sub-agent's prompt; sub-agents don't re-bootstrap the whole vault (a full read of a big note Ã— N agents is how token cost explodes). Verify a file/flag quoted in a note still exists before acting on it (memory can be stale). Rationale and numbers: [ADR-0018](docs/adr/0018-multi-agent-token-efficiency.md).
